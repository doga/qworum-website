Qworum — The UI layer for microservices architectures

Tweets by TwitterDev

Summary

Qworum is a foundational Web technology that makes Web-based user interfaces fully compatible with microservices architectures.

To this end Qworum defines a new type of microservice / Web API: the Qworum service. Qworum services are interactive microservices that serve as building blocks for modular and distributed Web applications. Qworum's browser technology is implemented by the Qworum browser extension.

While micro frontends are also used for the same purpose of breaking Web UIs into manageable pieces, Qworum offers a complementary approach with features that cannot be matched by micro frontends. Here is a brief comparison (see below for a more detailed comparison with micro frontends):

  • Qworum enables unbounded composability and cross-organisational applicability.
  • Whereas the UIs of micro frontends can only occupy a predefined display area within a web page, Qworum services can use a sequence of full-page UIs for interacting with the end-user.
  • Whereas micro frontends are Web page parts that are implemented with existing Web technologies, Qworum defines an entirely novel foundational Web technology and Web browser capability targeting microservices architectures specifically.

1. Specification

You can either directly jump to the Qworum specification, or first read the following in order to get some context about what Qworum is trying to achieve.

2. Solving the UI-monolith bottleneck for microservices architectures

2.1. What are microservices architectures?

Microservices architectures enable rapid software delivery by allowing different parts of a software system to be developed, deployed and updated in parallel by different teams within an organization and also across organizations.

Microservices architectures organise developer teams around vertical slices of software systems rather than the horizontal UI, business logic, and data layers. In order words, microservice teams are responsible for all of these 3 layers within the boundaries of a particular slice.

2.2. What is wrong with monolithic user interfaces?

For microservices architectures, having a monolithic component at the UI layer represents a bottleneck for the software development lifecycle, because the team responsible for the UI monolith cannot scale with the number of microservices.

2.3. Are micro frontends the right solution?

Today, the only solution that software developers have at their disposal for breaking UI monoliths into smaller parts involves micro frontends.

Micro frontends are UI components that are plugged into dashboard-style web pages for building web applications. In this case web applications become simple containers for micro frontends.

Micro frontends are implemented using various technologies such as React.js components, web components or iframes, none of which are specific to microservices architectures.

Besides being only suitable for dashboard-style applications, other limiting factors for micro frontends include:

  • Bounding context — Micro-frontend solutions are typically only usable within a particular context rather than being universally applicable. This limiting context can be a particular organization (which precludes cross-organization applications), a particular web application, or a particular frontend framework such as React.js or Angular for example.
  • Data exchange — The data exchange mechanism between a micro-frontend and its container is ad hoc at best. For example, if a micro-frontend is implemented as a React.js component, then the micro-frontend can use React.js's built-in mechanism for receiving data from its container. On the other hand, how and if a React.js component can return a result to its container is an implementation detail that needs to be resolved on a case by case basis. This is because micro frontends are conceived as UI components rather than full-blown services.
  • Nesting — Micro frontends typically cannot be nested beyond 1 or 2 levels, because the allocated screen area becomes significantly smaller with each nesting level.

2.3. Qworum

Qworum is a UI technology that was made specifically for microservices architectures. In contrast to micro frontends, which are parts of web pages, Qworum takes the web-API approach to user interaction.

In effect, Qworum defines a new type of web API where each end-point can receive data, interact with the end-user if needed, and then return a result. Note that Qworum services are not components embedded in web pages. Rather, they offer full-page UIs to end-users.

Also, in contrast to micro-frontend solutions where there are two types of entities (micro frontends and container applications), with Qworum everything is a Qworum service, including applications.

So, while Qworum complements micro frontends rather than making them obsolete, by the same token Qworum does not have many of the limitations that micro frontends are burdened with:

  • Bounding context — Qworum is a web-scale microservices platform that is usable both within organizations and across organizations. Qworum is also independent from any particular application or web framework.
  • Data exchange — Qworum defines a standard mechanism for exchanging data between Qworum services and their callers. This obviates the need for developing ad hoc data exchange solutions that are specific to a given application, framework or organization.
  • Nesting — Qworum allows for unbounded nesting levels.

3. A platform for distributed web applications

There are currently two different Webs that exist on the World Wide Web:

  • The Document Web: This the Web that most Web developers will be familiar with. The primary type of content is the HTML page, and the primary content consumers are human users rather than machines.
  • The Semantic Web: This the Web that is intended for machine consumption rather than human consumption. The primary type of content is RDF and its various encodings such as JSON-LD and Turtle.

Qworum adds another type of Web to the mix: the Service Web. Qworum is used for providing software rather than content. Qworum services are:

  • Interactive: They can interact with end-users through HTML pages.
  • Composable: They can call other Qworum services, and even themselves. And Qworum services can return data to their callers. A Qworum application, on the other hand, is simply a Qworum service that does not have a caller.

Note that although Qworum is distinct from the two previous Webs, it builds on them:

  • User interaction occurs through HTML pages. End-users will barely notice that they are using Qworum rather than the Document Web.
  • Qworum services can return semantic data to their callers, in addition to conventional data such as XML-wrapped JSON or any other XML data.

4. Demonstration — application spanning several DNS domains

This video shows an online shop calling a remote shopping cart service.

See this demo live at shop.demo.qworum.net. Be sure to install the Qworum extension for your desktop browser beforehand.

In this demonstration, the e-shop is a Qworum application and the shopping cart is a remote Qworum service. Thanks to Qworum, the shopping cart service integrates with the e-shop seamlessly, despite being hosted on another DNS domain. (The e-shop is hosted on shop.demo.qworum.net while the shopping cart is hosted on cart.demo.qworum.net)

Note that:

  • with Qworum the user experience is similar to the current Web, with the exception that the browser's back button will be disabled.
  • search engines can't peek inside Qworum services, so if search engine optimization (SEO) is important for you then you might want to carefully consider which parts of your application will be implemented as Qworum services. For enterprise and business applications SEO is generally not an issue, but consumer applications such as an online shop might want to only use remote Qworum services and limit the use of internal Qworum services to the parts that should remain out of reach for search engines.

5. A secure computing environment

The Qworum platform conforms to the same-origin policy that is the prevalent security mechanism for browser standards. In particular all phases of a given Qworum object are contained in the object's web origin. This ensures that data belonging to one web origin is unable to accidentally leak to another origin.

6. Use cases

Qworum is broadly applicable for all web applications, whether it be an e-commerce application or an enterprise application.

7. Enabling Qworum for your domain

Enabling Qworum for your internet or intranet domain requires a subscription. Qworum subscriptions are available for purchase to companies and individuals worldwide.

Pricing:

  • 400 USD per month or 4,000 USD per year. This covers up to 5 subdomains of the subscribed domain plus the subscribed domain itself. (If your company's domain is xyz.example and a subscription is requested for a.xyz.example, then the subscription will cover websites hosted on domains such as a.xyz.example, b.a.xyz.example and e.d.c.a.xyz.example. Subscriptions cover the standard TCP ports for both the HTTPS and the HTTP protocols.)
  • The number of subscribed subdomains can be increased for 80 USD per month per additional subdomain or 800 USD per year per additional subdomain.

Simply purchase your domain subscription by sending an email to info@qworum.net. Please make sure to include the following information:

  • Company name.
  • Company address.
  • Company's email address.
  • Company's DNS domain or subdomain for which the domain subscription is being requested.
  • The subscription renewal frequency (monthly or yearly).
  • Optionally, the number of additional subdomains to be included in the subscription.

You will receive an invoice by email. Your domain subscription will be activated within 7 workdays after payment (more precisely, 3 workdays + the time that Chrome Web Store takes for publishing the updated Qworum browser extension, which is generally less than 3 workdays). Your subscription period will start when the Qworum browser extension is enabled for your domain. You will get an email confirmation.

If you'd like to try Qworum first, Qworum is FREE to use on your local machine. Please refer to the developer resources section.

8. Developer Resources

  • Browser extension : Qworum applications must make sure that the user has installed and activated the Qworum extension. The demo linked above shows how this is done.
  • JavaScript library for Web Pages (API documentation) : With this JavaScript ES module, websites can use the advanced browser functionality that is provided by Qworum's browser extension.
  • Demo project 1: This GitHub repository contains the code base for the demonstration shown above. This project uses the JSON format for data that is received and returned by service end-points, and also for storing session data in the browser.
  • Demo project 2: This GitHub repository contains the code base for the demonstration shown above. This project uses semantic data instead of JSON for data that is received and returned by service end-points, and also for storing session data in the browser.
  • Demo in pseudocode: This is a TypeScript pseudocode that describes the demonstration. This is an easy way of seeing an overview of how the demo is structured.
  • Visual Studio Code extension: Provides XML snippets for writing static Qworum scripts. This VS Code extension is automatically activated for files that have the .qrm.xml extension.
  • Google group: Visit this group for feature requests, bug reports and general discussion.

Note that, although you need to contact us in order to use Qworum on your Internet website, you can develop Qworum-enabled websites locally right now. The web origins that are enabled by default are:

  • http://localhost:8080, http://27.0.0.1:8080 (TCP port 8080 is often used in the Java world)
  • http://localhost:3000, http://27.0.0.1:3000 (port 3000 is often used in Deno or Node.js projects)
  • http://localhost:5500, http://27.0.0.1:5500 (5500 is the default port for VS Code's Live Server extension)
  • http://localhost:5501, http://27.0.0.1:5501 (alternative Live Server port)
  • http://localhost:5502, http://27.0.0.1:5502 (alternative Live Server port)

9. Privacy Policy

Qworum respects the privacy of end-users. Qworum's browser extension only has access to data that websites send it, and this data is deleted when you close the website's tab. Your data does not leave your computer.
