Qworum — The Service Web

Qworum is:

  • a module system for Web applications, where modules may be located anywhere on the Web.
  • a distributed Web application technology.
  • a platform for interactive microservices; Qworum solves the "UI-monolith problem" for microservice architectures through services that are microservices which bring their own full-page micro-frontends.
  • the Service Web that complements the HTML-based Document Web and the RDF-based Semantic Web.
  • an upgrade to the World Wide Web for improved application support.
  • an advanced Web browser feature that websites can use.

Qworum is suitable for both:

  • Jamstack frontends that are made up of static files, and
  • the more legacy Web frontends that are dynamically generated on the server for each user.

1. Specification

You can either directly jump to the Qworum specification, or first read the following in order to get some context about what Qworum is trying to achieve.

2. Solving the UI bottleneck for microservice architectures

Microservice architectures enable rapid software delivery by allowing different parts of a software system to be developed, deployed and updated in parallel by different teams within an organization and also across organizations.

Microservice architectures organise developer teams around vertical slices of software systems rather than the horizontal UI, business logic, and data layers. In order words, microservice teams are responsible for all of these 3 layers within the boundaries of a particular slice.

2.1. What Qworum is solving

One problem that was unsolved until Qworum is the UI monolith issue for microservice architectures.

Indeed, without Qworum current microservice architectures are unable to slice the UI layer into microservices completely. Web applications still must have a monolithic component at the UI layer. This monolithic component acts as a container for the different microfrontends which constitute parts of the monolith's web pages.

Having a monolithic component at the UI layer represents a bottleneck for the software development and delivery processes, because the team responsible for the UI monolith cannot scale with the number of microservices. In addition, microfrontends are frequently tied to a particular web application or frontend framework, which limits their reusability. Also, the mechanisms by which microfrontends exchange data with their container is ad hoc at best.

The interactive service concept that Qworum defines solves all above-mentioned issues. Qworum services receive and return data through a well-defined mechanism, and they can interact with the end-user during their execution. Applications are the same as the services they are calling, in the sense that both types of entities are implemented as Qworum services. The only difference between an application and other services is that applications frequently run indefinitely without returning a result, and they are made for being directly accessed by the end-user rather than being called by another service.

Qworum is also solving the nesting issue that the current microfrontend solutions are facing. Indeed, microfrontends frequently do not contain other microfrontends nested in them, because the screen area allocated to nested elements quickly becomes vanishingly small.

In contrast, the nesting depth of Qworum services is unbounded, because Qworum allocates a full page to each Qworum service regardless of the nesting depth.

3. A platform for distributed web applications

There are currently two different Webs that exist on the World Wide Web:

  • The Document Web: This the Web that most Web developers will be familiar with. The primary type of content is the HTML page, and the primary content consumers are human users rather than machines.
  • The Semantic Web: This the Web that is intended for machine consumption rather than human consumption. The primary type of content is RDF and its various encodings such as JSON-LD and Turtle.

Qworum adds another type of Web to the mix: the Service Web. Qworum is used for providing software rather than content. Qworum services are:

  • Interactive: They can interact with end-users through HTML pages.
  • Composable: They can call other Qworum services, and even themselves. And Qworum services can return data to their callers. A Qworum application, on the other hand, is simply a Qworum service that does not have a caller.

Note that although Qworum is distinct from the two previous Webs, it builds on them:

  • User interaction occurs through HTML pages. End-users will barely notice that they are using Qworum rather than the Document Web.
  • Qworum services can return semantic data to their callers, in addition to conventional data such as XML-wrapped JSON or any other XML data.

In this demonstration, the e-shop is a Qworum application and the shopping cart is a remote Qworum service.

Note that:

  • with Qworum the user experience is similar to the current Web, with the exception that the browser's back button will be disabled.
  • search engines can't peek inside Qworum services, so if search engine optimization (SEO) is important for you then you might want to carefully consider which parts of your application will be implemented as Qworum services. For enterprise and business applications SEO is generally not an issue, but consumer applications such as an online shop might want to only use remote Qworum services and limit the use of internal Qworum services to the parts that should remain out of reach for search engines.

5. A secure computing environment

The Qworum platform conforms to the same-origin policy that is the prevalent security mechanism for browser standards. In particular all phases of a given Qworum object are contained in the object's web origin. This ensures that data belonging to one web origin is unable to accidentally leak to another origin.

6. Use cases

The main use cases for Qworum are:

  • Application integration: Different Web-based applications can provide Qworum services to each other and provide a seamless user experience that combines both applications.
  • Productivity gains when developing web applications: An interactive service needs to be implemented and made available on the Web only once. All Web applications can then leverage that service and spare countless development hours.

Qworum is widely applicable. If an online service is interactive, then it can benefit from using Qworum. Here is a foretaste of the types of services that Qworum can enable:

  • User authentication
  • Payment processing
  • Collection management — shopping cart, addressbook/contacts, movies, files, customer information, supplier information, …

7. How to enable Qworum for your website

In order to enable Qworum for your website, simply send us an email by using this link. Be sure to set the URL in the subject line to your own website URL. You can leave the email body empty, we will not read it.

If you don't enable Qworum for your website, you will get JavaScript run-time errors on your website if you try to use Qworum.

Enabling Qworum for your website is FREE and always will be.

8. Developer Resources

  • Browser extension : Qworum applications must make sure that the user has installed and activated the Qworum extension. The demo linked above shows how this is done.
  • JavaScript library for Web Pages : With this JavaScript ES module, websites can use the advanced browser functionality that is provided by Qworum's browser extension.
  • Demo project 1: This GitHub repository contains the code base for the demonstration shown above. This project uses the JSON format for data that is received and returned by service end-points, and also for storing session data in the browser.
  • Demo project 2: This GitHub repository contains the code base for the demonstration shown above. This project uses semantic data instead of JSON for data that is received and returned by service end-points, and also for storing session data in the browser.
  • Visual Studio Code extension: Provides XML snippets for writing static Qworum scripts. This VS Code extension is automatically activated for files that have the .qrm.xml extension.

9. Privacy Policy

Qworum respects the privacy of end-users. Qworum's browser extension only has access to data that websites send it, and this data is deleted when you close the website's tab. Your data does not leave your computer.
